# Swipey — описание продукта и технический путеводитель

> Документ отражает фактическое состояние проекта на 16 июля 2026 года. Он предназначен как точка входа для разработчиков и AI-агентов: что делает приложение, как устроен код, где хранится состояние, какие сценарии уже работают и что пока является заглушкой.

## 1. Кратко о приложении

Swipey — iOS-приложение для быстрой очистки медиатеки. Пользователь просматривает фотографии и видео по одному и принимает решение привычным свайпом:

- свайп вправо — оставить медиафайл;
- свайп влево — добавить медиафайл в очередь на удаление;
- кнопка «Отменить» — отменить последнее решение;
- кнопка «Удалить N» — запросить подтверждение и удалить отмеченные объекты из системной медиатеки.

Удаление не происходит сразу после свайпа влево. Сначала идентификатор попадает во внутреннюю очередь, а реальное удаление выполняется только после отдельного подтверждения пользователя через системный PhotoKit API.

Текущая версия является ранним продуктовым прототипом: основной swipe-flow, работа с фото и видео, разрешения, локальный дневной лимит и экран paywall уже существуют. Реальные покупки, подписки, история удалений, альбомы, аналитика и тесты пока не реализованы.

## 3. Пользовательский сценарий

### 3.1. Первый запуск

1. `SwipeyApp` создаёт `Store<SwipeFeature.State, SwipeFeature.Action>`.
2. `ContentView` отправляет `.view(.task)`.
3. Reducer запрашивает read/write-доступ к медиатеке.
4. При разрешённом или ограниченном доступе загружаются все доступные фото и видео.
5. Объекты сортируются от новых к старым по `creationDate`.
6. Загружается оставшееся число свайпов и прогреваются следующие миниатюры.
7. Открывается основной экран.

### 3.2. Состояния доступа

- `notDetermined` — показывается экран загрузки, пока система ожидает решение пользователя;
- `authorized` — показывается основной swipe-интерфейс; сюда также отображается PhotoKit-статус `.limited`;
- `denied` — показывается объяснение с кнопками перехода в Settings и повторной проверки.

Для доступа используются ключи Info.plist, генерируемые из build settings:

- `NSPhotoLibraryUsageDescription` — просмотр и очистка галереи;
- `NSPhotoLibraryAddUsageDescription` — удаление выбранных фотографий.

### 3.3. Просмотр карточки

Карточка занимает центральную часть экрана. Фото показывается целиком через `scaledToFit`. Для видео сначала отображается thumbnail, затем создаётся `AVPlayerItem`; пользователь может запустить и приостановить воспроизведение.

Во время drag-жеста карточка:

- смещается и слегка поворачивается;
- становится немного прозрачнее;
- показывает зелёный badge «ОСТАВИТЬ» справа или красный «УДАЛИТЬ» слева;
- считается свайпнутой при горизонтальном смещении более 120 pt.

После завершённой анимации UI отправляет reducer-у направление свайпа. Каждый успешный свайп расходует одну единицу дневного лимита независимо от направления.

### 3.4. Отмена

История хранит `assetID` и направление каждого завершённого свайпа. Отмена:

- возвращает `currentIndex` на одну позицию назад;
- удаляет объект из очереди удаления, если отменён свайп влево;
- не возвращает потраченный дневной свайп.

Поддерживается только последовательная отмена с конца истории. Shake-to-undo отсутствует.

### 3.5. Удаление

После нажатия «Удалить N» приложение показывает alert. При подтверждении:

1. вычисляет, сколько удаляемых объектов находится до текущего индекса;
2. вызывает `PhotoManager.deleteAssets(with:)`;
3. PhotoKit показывает системное подтверждение изменения медиатеки;
4. успешно удалённые объекты убираются из локальных массивов и кэшей;
5. индекс корректируется, очередь очищается, счётчик «Удалено» увеличивается.

При ошибке показывается alert: «Не удалось удалить медиафайлы…».

Важно: метаданные удалённых объектов не сохраняются, собственного восстановления нет. Возможность восстановления определяется системной папкой «Недавно удалённые» приложения Photos.

### 3.6. Лимит и paywall

`SwipeLimiter` хранит число использованных свайпов и дату последнего сброса в `UserDefaults`:

- `swipey.swipesUsed`;
- `swipey.lastResetDate`.

Счётчик сбрасывается при первом обращении в новый календарный день по локальному календарю устройства. Фактический лимит сейчас равен **5000 свайпов в день**. Это явно временное значение: README описывает будущий бесплатный лимит 20.

Когда остаток достигает нуля, открывается полноэкранный paywall. Кнопка «Оформить подписку» пока ничего не делает; SDK покупок и состояние entitlement отсутствуют. Закрытие paywall возвращает пользователя назад, но не снимает лимит.

## 4. Архитектура

```text
SwipeyApp
  └── Store<SwipeFeature>
       └── ContentView
            ├── loadingView
            ├── PermissionDeniedView
            ├── SwipeView
            │    └── AssetCardMediaView
            └── PaywallView

SwipeFeature
  ├── PhotoLibraryClient ──> PhotoManager ──> PhotoKit / AVFoundation
  └── SwipeLimiterClient ──> SwipeLimiter ──> UserDefaults
```

Проект находится в промежуточном состоянии миграции на TCA. Бизнес-состояние свайпов уже сосредоточено в `SwipeFeature`, а доступ к PhotoKit и лимиту частично завернут в dependency clients. Однако загрузка контента карточки и подтверждённое удаление всё ещё обращаются к singleton `PhotoManager.shared` напрямую из SwiftUI.

### 4.1. TCA State

`SwipeFeature.State` содержит:

| Поле | Назначение |
| --- | --- |
| `isLoading` | Идёт ли стартовая загрузка или повторная проверка доступа |
| `currentIndex` | Индекс текущей карточки в `assets` |
| `assets` | Лёгкие доменные модели доступных фото и видео |
| `queuedForDeletion` | ID объектов, отмеченных свайпом влево |
| `deletedCount` | Число объектов, удалённых за текущую сессию |
| `accessState` | Состояние разрешения медиатеки |
| `errorMessage` | Текст общей ошибки для alert |
| `showDeleteConfirmation` | Нужно ли показать подтверждение удаления |
| `isPaywallPresented` | Открыт ли paywall |
| `swipeHistory` | История решений для undo |
| `didBootstrap` | Защита от повторного выполнения стартовой `.task` |
| `remainingSwipes` | Остаток локального дневного лимита |

Вычисляемые свойства `currentAsset`, `hasAssets`, `canSwipe`, `totalAssetsCount` и `canUndo` используются представлениями.

### 4.2. TCA Actions

Пользовательские события находятся внутри `Action.view(View)`: запуск, retry доступа, завершение свайпа, undo, запрос и закрытие подтверждения удаления, открытие/закрытие paywall и закрытие ошибки.

Внутренние response-actions:

- `bootstrapResponse` и `authorizationRefreshed` применяют доступ, список assets и лимит;
- `swipeProcessed` фиксирует успешно израсходованный свайп;
- `deleteQueuedAssetsResponse` обновляет состояние после PhotoKit-операции.

`View.deleteConfirmedTapped` объявлен, но reducer сейчас возвращает `.none`; UI его не отправляет и выполняет удаление самостоятельно.

### 4.3. Доменные модели

- `MediaAsset` — `id` PhotoKit-объекта и тип `.photo`/`.video`; тяжёлый `PHAsset` намеренно не хранится в TCA State;
- `SwipeAction` — ID и направление решения для undo;
- `SwipeDirection` — `.left` или `.right`;
- `LibraryAccessState` — упрощённое состояние доступа.

`MediaAsset` поддерживает `Codable`, хотя сейчас не сериализуется.

## 5. Работа с медиатекой

### 5.1. PhotoManager

`PhotoManager` — `@MainActor` singleton и фактический адаптер над PhotoKit. Он:

- запрашивает и обновляет authorization status;
- загружает изображения и видео;
- хранит соответствие `localIdentifier -> PHAsset`;
- сортирует медиатеку от новой к старой;
- получает thumbnail с разрешённой сетевой загрузкой из iCloud;
- подготавливает ближайшие thumbnails через `PHCachingImageManager`;
- получает `AVPlayerItem` для видео;
- удаляет выбранные assets;
- чистит собственные кэши после удаления.

Кэш изображений и видео основан на `NSCache`. Явного лимита памяти и явной остановки PhotoKit preheating сейчас нет.

### 5.2. PhotoLibraryClient

TCA dependency предоставляет операции запроса доступа, обновления статуса, загрузки assets, preheating, удаления и получения thumbnail. Live-реализация делегирует `PhotoManager`.

Не все возможности manager-а представлены клиентом: получение `AVPlayerItem` отсутствует. Поэтому `AssetCardMediaView` обращается к singleton напрямую.

### 5.3. Limited Photos Access

Статус `.limited` считается авторизованным. Приложение видит только выбранные пользователем объекты. Отдельного UI для изменения limited-набора нет.

## 6. UI-слой

### `SwipeyApp.swift`

Точка входа. Создаёт единственный `SwipeFeature` store без отдельного корневого `AppFeature`.

### `ContentView.swift`

Корневой router на основе state: загрузка, основной экран или отказ в доступе. Здесь же находятся presentation bindings для paywall и error alert и переход в системные Settings.

### `SwipeView.swift`

Главный экран, drag-анимация, haptic feedback, верхняя статистика, undo/delete-кнопки, empty state и alert удаления. Визуальное состояние жеста (`dragOffset`, `cardOpacity`) локально в SwiftUI, что правильно отделяет эфемерную анимацию от бизнес-state.

### `AssetCardMediaView.swift`

Загружает и показывает текущий медиафайл. Для видео управляет локальным `AVPlayer`, останавливая его при смене карточки или исчезновении view.

### `PermissionDeniedView.swift`

Объясняет необходимость доступа, позволяет открыть Settings и повторно проверить разрешение.

### `PaywallView.swift`

Визуальная заглушка полноэкранного paywall. Реальной покупки, цены, условий, restore и ссылок на документы нет.

## 8. Потоки данных

### Стартовая загрузка

```text
ContentView.task
  -> SwipeFeature.view.task
  -> PhotoLibraryClient.requestAccessIfNeeded
  -> PhotoLibraryClient.loadAssets
  -> SwipeLimiterClient.remaining
  -> PhotoLibraryClient.preheatThumbnails
  -> bootstrapResponse
  -> обновление UI
```

### Свайп

```text
DragGesture > 120 pt
  -> локальная анимация карточки
  -> view.swipeCompleted(direction)
  -> SwipeLimiterClient.consumeSwipe
  -> swipeProcessed
  -> очередь удаления / история / currentIndex
  -> preheat следующих карточек
```

### Удаление

```text
Кнопка «Удалить N»
  -> queueDeletionTapped
  -> alert
  -> прямой вызов PhotoManager из SwipeView
  -> системное подтверждение PhotoKit
  -> deleteQueuedAssetsResponse
  -> очистка TCA state
```

## 10. Известные ограничения и технический долг

1. **Paywall — заглушка.** Кнопка подписки пустая, RevenueCat/StoreKit не подключены, premium entitlement отсутствует.
2. **Лимит временный.** В коде 5000, в продуктовой идее README — 20 бесплатных свайпов в день.
3. **Удаление обходит reducer.** Асинхронная операция живёт в `SwipeView`, хотя `PhotoLibraryClient.deleteAssets` уже существует.
4. **Видео обходит dependency layer.** `playerItem` берётся напрямую у `PhotoManager.shared`.
5. **Undo не возвращает лимит.** Пользователь платит свайпом даже после отмены решения.
6. **Нет persistence прогресса.** После перезапуска список начинается сначала; очередь, индекс, история и `deletedCount` не сохраняются.
7. **Нет синхронизации изменений медиатеки.** Приложение не подписано на `PHPhotoLibraryChangeObserver`, поэтому внешние изменения во время сессии могут не отразиться.
8. **Нет отмены image requests.** Быстрая смена карточек не отменяет незавершённые PhotoKit-запросы.
9. **Нет тестов.** Reducer и dependency clients тестопригодны, но test target отсутствует.
10. **Нет фильтров.** Фото и видео смешаны; нельзя выбрать тип, альбом, период или размер.
11. **Нет deleted history.** Сохраняется только сессионный числовой счётчик.
12. **Нет полноценной локализации и accessibility-аудита.** Строки зашиты в Swift-файлы.
13. **Empty state формулируется шире limited-доступа.** «Все доступные» может означать только разрешённую пользователем часть медиатеки.
14. **Нет отдельного app-level reducer.** Навигация и entitlement по мере роста потребуют `AppFeature` или аналогичного корневого домена.
15. **README расходится с кодом.** Он всё ещё называет TCA и video support будущими этапами.

## 11. Планируемое продуктовое развитие

Согласно существующему README, целевой продукт предполагает:

- бесплатный лимит 20 фото-свайпов в день;
- безлимитные свайпы для подписчиков;
- premium-доступ к очистке видео;
- историю удалённых объектов;
- очистку по отдельным альбомам;
- RevenueCat как источник entitlement и offerings;
- purchase и restore purchase;
- shake-to-undo;
- аналитику ключевых этапов воронки;
- расширенное reducer-тестирование.

Это roadmap, а не описание текущей реализации. Перед разработкой конкретной функции следует сверять продуктовые решения: должен ли undo возвращать квоту, что именно хранить в истории, разрешён ли preview видео бесплатным пользователям и как показывать locked-функции.

## 12. Рекомендуемое направление рефакторинга

Ближайший технически логичный шаг — закончить границу TCA:

1. перенести подтверждённое удаление в эффект `SwipeFeature` через `photoLibraryClient.deleteAssets`;
2. добавить `playerItem` в dependency client или выделить media content client;
3. инъецировать clock/calendar и storage в лимитер для детерминированных тестов;
4. создать test target и покрыть bootstrap, left/right swipe, undo, limit reached и delete success/failure;
5. добавить app-level entitlement state до подключения реальных покупок;
6. обновить README так, чтобы roadmap был явно отделён от implemented features.

## 13. Как запускать и проверять

1. Открыть `Swipey.xcodeproj` в Xcode.
2. Выбрать схему `Swipey` и iOS 18+ simulator/device.
3. Дождаться разрешения Swift Package Manager зависимости TCA.
4. Собрать и запустить приложение.
5. Разрешить доступ ко всей медиатеке или выбранным объектам.

Для реалистичной проверки фото, видео, iCloud-загрузки, limited access и удаления предпочтительно физическое устройство. Удаление является реальной операцией над пользовательской медиатекой и требует внимательного тестового набора данных.

Рекомендуемый smoke-check:

- первый запуск и каждый вариант разрешения;
- свайп вправо и влево;
- undo для обоих направлений;
- удаление одного и нескольких объектов;
- отмена системного удаления;
- просмотр локального и iCloud-видео;
- достижение лимита с временно уменьшенным `dailyLimit`;
- перезапуск в тот же день и после смены календарного дня.

## 14. Правила для дальнейшей работы с проектом

- Источником истины о swipe-flow считать `SwipeFeature.State` и reducer.
- Не хранить `PHAsset`, `UIImage` или `AVPlayerItem` в TCA State; использовать стабильные local identifiers и dependency layer.
- Эфемерную анимацию жеста оставлять в SwiftUI, бизнес-решения направлять actions в reducer.
- Любую необратимую операцию PhotoKit выполнять только после явного подтверждения пользователя.
- Не считать paywall рабочим до появления purchase, restore, entitlement refresh, error states и юридических ссылок.
- При изменении лимита учитывать календарный сброс, undo и подписочный статус вместе.
- Обновлять этот документ после существенных изменений архитектуры или функциональности.

## 15. Быстрый словарь

- **Asset** — фото или видео из PhotoKit.
- **Keep** — свайп вправо; объект остаётся в медиатеке.
- **Queue for deletion** — свайп влево; объект лишь помечается внутри приложения.
- **Delete** — подтверждённая системная операция удаления через PhotoKit.
- **Swipe limit** — число решений по карточкам, а не число удалений.
- **Preheating** — предварительная подготовка следующих thumbnail для более плавного интерфейса.
- **Entitlement** — будущее подтверждённое право пользователя на premium-функции; сейчас отсутствует.
